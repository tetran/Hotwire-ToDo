# Model Decomposition Patterns

ActiveRecord モデルが肥大化した際に適用するパターンのガイドライン。
各パターンには「いつ使うか」と「いつ使わないか」を明記する。

## 1. Value Object

**何か:** ビジネスロジックを持つイミュータブルな PORO。等価性は値で決まる（identity ではない）。

**いつ使うか:**
- 複数の属性がセットで意味を持ち、計算ロジックを伴う場合
- 同じ属性グループに対する操作が複数のクラスに散らばっている場合
- concern + service で扱っているロジックが、実は1つの「値」を表している場合

**いつ使わないか:**
- 属性が1〜2個で計算ロジックがほぼ無い場合（過剰抽象）
- DB 永続化が必要な場合（それは Model の責務）

**実装規約:**
- `app/models/` に配置（`app/values/` は作らない）
- コンストラクタで `freeze` してイミュータブルにする
- `==`, `eql?`, `hash` を実装する
- DB カラムは元のモデルに残し、VO はそこから構築する
- バリデーションは元のモデルに残す（ActiveModel `errors` のため）
- `Comparable` は `<=>` を実装する場合のみ include する

**実例:** `RecurrenceRule`（#277）— TaskSeries のスケジューリング属性 6 個 + IceCube 計算 + 人間可読ラベル生成を統合

## 2. Policy Object

**何か:** 認可・権限判定ロジックをカプセル化する PORO。

**いつ使うか:**
- モデルに認可メソッドが 3 個以上あり、ドメインロジックと混在している場合
- 認可ロジックが複数のリソースタイプにまたがる場合
- テストで認可ロジックだけを独立して検証したい場合

**いつ使わないか:**
- 認可メソッドが 1〜2 個で単純な場合（移動コストに見合わない）
- Pundit 等の認可 gem を導入する場合（gem の規約に従う）

**実装規約:**
- `app/models/` に配置（`app/policies/` は Pundit 規約のため使わない）
- 元のモデルから `delegate` で委譲し、既存インターフェースを維持する
- クエリ結果をキャッシュしない（毎回 DB に問い合わせる）
- キャッシュが必要になった場合は invalidation 機構とセットで導入する

**実例:** `AdminPolicy`（#276）— User の認可メソッド 6 個を抽出、50+ 箇所の呼び出し元は変更不要

## 3. Concern（コールバック/副作用の分離）

**何か:** モデルのコールバックや副作用ロジックを ActiveSupport::Concern に分離する。

**いつ使うか:**
- コールバック + private メソッドのセットが明確な責務を形成している場合
- `broadcast_*`, `notify_*`, `sync_*` のような副作用ロジックがドメインロジックと混在している場合
- 分離するメソッド群が Turbo/ActionCable 等のフレームワークインスタンスメソッドに依存している場合

**いつ使わないか:**
- 外部 PORO に切り出せる場合（concern より PORO が優先）
- メソッドが 1〜2 個しかない場合（concern にするほどでもない）
- 複数モデルで共有しない限り、汎用的な名前の concern は作らない

**実装規約:**
- `app/models/concerns/{model_name}/` に配置
- ファイル先頭は `class ModelName`（`module ModelName` ではない — クラスの再オープンで名前衝突を回避）
- `included do ... end` ブロックで callback を宣言し、条件（`:if`, `:unless`）も完全に再現する
- 既存パターン参照: `Task::Broadcasting`

## 判断フローチャート

```
モデルが肥大化している
  │
  ├─ ロジックがフレームワークのインスタンスメソッドに依存？
  │   └─ Yes → Concern で分離
  │
  ├─ 複数属性 + 計算ロジック が1つの「値」を表す？
  │   └─ Yes → Value Object を抽出
  │
  ├─ 認可/権限判定ロジックが集中？
  │   └─ Yes → Policy Object を抽出
  │
  ├─ 複数モデルをまたぐ操作やトランザクション？
  │   └─ Yes → Service Object（既存の app/services/ パターン）
  │
  └─ 上記に当てはまらない → まだ分離不要。痛みが出てから対処する
```
